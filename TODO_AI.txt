----------------------------------------------------------------------
GUIDELINES FOR AI AGENTS
----------------------------------------------------------------------

1. Verify state before acting. Items may already be done.
2. Fix root causes, not symptoms.
3. If you find bugs not listed here, fix them and note them in the
   commit message.
4. Don't force architectural changes unless they unblock the current
   task.
5. Pull → work → commit → push is the default rhythm.
   - Commit each file change as it lands unless a single logical
     unit genuinely spans multiple files.
   - `git push` after every commit. No local-only commits.
   - If you are mid-refactor and something breaks, commit the WIP
     under a `wip:` prefix and push — never sit on broken work.
6. Run `python smoke_test.py` (~5 s) before every commit. Run
   `python -m crawlers wave sanity` for the slightly stronger gate.
   Full `python -m crawlers wave all` only at release boundaries —
   running it between every commit wastes minutes on output that
   does not change.
7. Batch terminal permissions up front. Front-load shell operations
   into long combined commands rather than drip-feeding many small
   calls. List every command you will need at the top of the reply
   when the plan is predictable.
8. Read PROJECT.md §11 (Reusable abstractions) and §12 (Testing)
   before adding new code. Pick the right helper off that list
   rather than inventing a parallel approach.
9. Claude vs. Ollama: Claude handles planning, architecture, risky
   debugging, auth/permissions, multi-file refactors, and schema
   decisions. Ollama is for small bounded tasks (single-file edits,
   boilerplate, repetitive cleanup, summaries, first-pass drafts).
   Ollama must NEVER touch security-critical logic, the permission
   model, or do broad stateful refactors. See README.md §"Claude +
   Ollama workflow" for the bridge details.

DESIGN PHILOSOPHY (load-bearing — read every session)
----------------------------------------------------------------------

Apple / Jony Ive / Ferrari. Every element earns its place.

  - **Tiles, not pages.** Each concern is a self-contained widget
    tile on a fluid grid. No mixed concerns inside one card.
  - **Pagination, never scroll.** Long lists use `paginated_pane`.
    No `overflow: auto` inside content panes.
  - **Macros over markup.** If a pattern shows up twice, it becomes
    a macro. Templates compose; they do not duplicate.
  - **One queue, one card.** A request is the same object everywhere
    it appears — same identity, same status block, same actions.
  - **Visibility is sliced server-side first.** `data-vis` is a
    safety net, not the gate.
  - **Empty space is content.** Cramped is the failure mode, not
    bold.

Reference implementation: `templates/instrument_detail.html` — 10
tiles on a 6-column fluid grid. CSS at `static/styles.css` under
"INSTRUMENT DETAIL — TILE ARCHITECTURE". This is the gold standard.
Match its rhythm.

v1.3.0 — HARDENING COMPLETE
======================================================================

Four changes that close the security / correctness story. None of
them touch the user-visible feature set; they make the existing
features unbreakable.

----------------------------------------------------------------------
1.3.0-a  Split request_detail() into per-action handlers      XL  NEXT
----------------------------------------------------------------------

The 685-line `request_detail()` view is the largest single function
in the codebase. It dispatches 14 actions: approve_step, reject_step,
schedule, admin_schedule_override, start, complete,
admin_complete_override, reject, mark_sample_submitted,
mark_sample_received, cancel, reassign_operator, and two more.

Plan:
  - Extract one private function per action: `_action_approve_step`,
    `_action_schedule`, `_action_complete`, etc.
  - Each takes `(user, sample_request, form_data)` and returns a
    `(flash_message, flash_category, redirect_url)` tuple.
  - The dispatch lives in `request_detail()`: a small dict mapping
    action name → handler function.
  - All status writes already go through `assert_status_transition()`,
    so no new state machine work needed.
  - Permission checks (`can_manage`, `can_operate`,
    `completion_locked`) stay at the top of `request_detail()` before
    dispatch, not in each handler.
  - Tests: visibility audit + populate crawl + the new
    test_status_transitions.py from 1.3.0-d (when it lands).

Files: `app.py` only.

----------------------------------------------------------------------
1.3.0-b  Flip CSRF enforcement on                              M
----------------------------------------------------------------------

The Flask-WTF machinery is in place but `WTF_CSRF_ENABLED` defaults
to off so the existing forms / tests / demo agents don't break.
Rolling enforcement out:

  1. Add `<input type="hidden" name="csrf_token"
     value="{{ csrf_token() }}">` to every `<form method="post">`
     in templates (~30 forms).
  2. Update test_visibility_audit.py, test_populate_crawl.py,
     smoke_test.py to fetch the meta tag and send the token.
  3. Default `LAB_SCHEDULER_CSRF=1` in start.sh.
  4. Re-run all three test scripts.

Files: every template with a POST form, the three test scripts,
start.sh, app.py (default).

----------------------------------------------------------------------
1.3.0-c  Server-side input validation everywhere               M
----------------------------------------------------------------------

Wrap every `int(request.form[…])` and `float(request.form[…])` in
`safe_int()` / `safe_float()`. Reject malformed input with a clean
toast instead of a 500. The toast system from v1.2.0 makes the
surfacing cheap. The pattern:

    raw = request.form.get(field)
    val = safe_int(raw, minimum=1, maximum=99)
    if val is None:
        flash("…", "error")
        return redirect(...)

Audit by grepping for `int(request.form` and `float(request.form`.
Roughly ~30 sites.

Files: `app.py` only.

----------------------------------------------------------------------
1.3.0-d  test_status_transitions.py                            S
----------------------------------------------------------------------

Persist the legal-pair exhaustive check as a real test file.
Exhaustively walks `REQUEST_STATUS_TRANSITIONS` and asserts every
legal pair succeeds + a sample of illegal pairs raises
`InvalidStatusTransition`. Adds it to the regression gates.

Files: `tests/test_status_transitions.py` (new), pre-push gate
docs.

======================================================================
v1.4.0 — BULK OPERATIONS
======================================================================

The bulk-actions tile on the queue exists but is currently a
placeholder. Wire the real actions:

  - bulk approve (advance approval step)
  - bulk assign operator
  - bulk schedule
  - bulk reject
  - bulk export

Each action runs per-row through the same permission gate the
single-row path uses; failures are reported in aggregate via the
toast system. The first tangible payoff from the tile architecture.

Files: `templates/schedule.html`, `app.py` (`schedule_bulk_actions`
route), `static/styles.css` (bulk-actions tile reveal animation).

======================================================================
v1.5.0 — SEARCH
======================================================================

SQLite FTS5 virtual table over `sample_name`, `description`,
`request_no`, and requester name. New search box on the queue and
dashboard; results link into the existing card view.

  - FTS5 ships with SQLite, so no new dependency.
  - Index rebuilt by `init_db()` if missing.
  - Trigger on `sample_requests` INSERT/UPDATE keeps it current.

Files: `app.py` (init_db, new search route), `templates/schedule.html`,
`templates/dashboard.html`.

Listed here so future agents do not spin them up speculatively.
Each one is held for a specific reason; un-hold it only when a real
user asks.

  - **Email notifications** — needs SMTP infra decision (relay vs
    Gmail app password vs SES). Wait until the production deployment
    has chosen its mail story.
  - **Cost / grant tracking + invoicing** — biggest item in the
    backlog and needs a per-instrument cost model first. No user yet.
  - **Per-instrument custom request fields** — speculative; nobody
    has asked. Build the second time someone hits the limit.
  - **Recurring / templated requests** — nice-to-have after search
    exists. v1.5.0 first.
  - **Mobile breakpoint pass** — desktop use is 95% of the load.
    Wait for a real bench operator complaint.
  - **Approval delegation** — wait for the first complaint about a
    travelling approver blocking the queue.
  - **Read-only `/api/v1` JSON blueprint** — build it against a real
    consumer, not in advance.
  - **Per-user dashboard customization** — gold-plating.
  - **Audit log viewer** — compliance checkbox. Build when audited.
  - **OAuth2 / SAML SSO** — couple to the actual MIT-WPU IdP, not a
    generic one. Wait for the production deployment.
  - **Multi-tenant** — only if PRISM is sold beyond MIT-WPU.

----------------------------------------------------------------------
NOT DOING (architectural decisions)
----------------------------------------------------------------------

Listed here so future agents do not spin them up speculatively.

  - Python 2 compatibility — no.
  - Server-side rendering of charts — Chart.js is fine.
  - Migrations framework (alembic / yoyo) — `init_db()` handles
    schema evolution adequately for SQLite.
  - Background job queue (celery / rq) — no async work justifies
    the dependency yet.
  - WebSockets — polling at the existing intervals is fine.
  - Build step / bundler / transpiler — vanilla JS + a single CSS
    file is the design rule.
