CATALYST — Active Plan
====================

Forward-looking only. The current release is **v1.3.0** — the
first stable release. The roadmap below covers v1.4.0 → v1.5.0
plus the held backlog and the in-flight UX polish stream.

For THE PHILOSOPHY read PHILOSOPHY.md (load-bearing, read every
session).
For the architecture spec read PROJECT.md.
For the version history read CHANGELOG.md.
For production deployment read DEPLOY.md.

----------------------------------------------------------------------
GUIDELINES FOR AI AGENTS
----------------------------------------------------------------------

1. Read PHILOSOPHY.md first, every session. It is load-bearing.
   Reject any change that violates it.
2. Verify state before acting. Items may already be done.
3. Fix root causes, not symptoms.
4. Hard attributes are locked (PHILOSOPHY.md §2). Changes to the
   data model, routes, roles, audit chain, or tile architecture
   require a major version bump and a `### Changed (BREAKING)`
   entry in CHANGELOG.md. Soft attributes (copy, placement,
   colours) drift freely between patch releases.
5. Every release on master is stable (PHILOSOPHY.md §3). No WIP
   on master. Experimental work lives on short-lived branches and
   merges only after `wave sanity` passes.
6. The website stays up. Deploys to the Mac mini are atomic:
   pull → smoke → `launchctl kickstart`. Never interrupt live
   users. See DEPLOY.md.
7. Pull → work → commit → push is the default rhythm.
   - Commit each file change as it lands unless a single logical
     unit genuinely spans multiple files.
   - `git push` after every commit. No local-only commits.
   - If you are mid-refactor and something breaks, commit the WIP
     under a `wip:` prefix on a branch — never sit on broken work
     on master.
8. Run `.venv/bin/python smoke_test.py` (~5 s) before every commit.
   Run `python -m crawlers wave sanity` for the slightly stronger
   gate. Full `wave all` only at release boundaries.
9. Batch terminal permissions up front. Front-load shell operations
   into long combined commands rather than drip-feeding many small
   calls. List every command you will need at the top of the reply
   when the plan is predictable.
10. Read PROJECT.md §11 (Reusable abstractions) and §12 (Testing)
    before adding new code. Pick the right helper off that list
    rather than inventing a parallel approach.
11. Demo data is never operational data (PHILOSOPHY.md §4). Demo
    only runs when `LAB_SCHEDULER_DEMO_MODE=1`. Production deploys
    on the Mac mini ship with it set to 0.
12. Any agent must be able to pick up where you left off from this
    file alone. No tribal knowledge.

v1.3.x PATCH STREAM — soft-attribute polish
======================================================================

Patch releases. Hard attributes untouched. Each item ships as a
separate patch release (1.3.1, 1.3.2, …) gated by `wave sanity`.

----------------------------------------------------------------------
1.3.1  safe_int / safe_float wrap                              M
----------------------------------------------------------------------

Wrap every `int(request.form[…])` and `float(request.form[…])` in
`safe_int()` / `safe_float()`. Reject malformed input with a clean
toast instead of a 500. The pattern:

    raw = request.form.get(field)
    val = safe_int(raw, minimum=1, maximum=99)
    if val is None:
        flash("…", "error")
        return redirect(...)

Audit by grepping for `int(request.form` and `float(request.form`.
Roughly ~30 sites.

Files: `app.py` only.

----------------------------------------------------------------------
1.3.2  Instrument page polish (Jony Ive / give the user what they
       want)                                                    M
----------------------------------------------------------------------

The instrument page is the gold standard. This patch sharpens the
remaining soft attributes.

  a. Remove visual-only header elements. Metadata lives in the
     metadata_grid tile, not duplicated in a decorative pane.
  b. L1 / D1 demo back-hover button. A margin hover affordance on
     the request detail page that returns the user to the page
     they came from (referrer-aware, falls back to /schedule).
  c. Accept / reject blob on the request info tile. Actions live
     where the user is looking — no tab nav, no modal.
  d. In-place metadata edit on the instrument page. Click a field,
     edit, blur → POST, append one entry to the instrument's
     event stream. No form dialog.
  e. Similar event panes for machines and jobs (currently only
     requests have a full activity_feed).
  f. Form control panel on the machine page: edit how the machine's
     request form travels through the system. Which faculty /
     operator group receives it, which metadata fields are
     required vs optional, which optional fields the admin can
     force.
  g. Sample approval explainer pane — default text mentioning how
     a sample travels through the approval pipeline, editable by
     admins, visible to every new user. Inline, not buried in a
     wiki.

Files: `templates/instrument_detail.html`,
`templates/request_detail.html`, `static/styles.css`, `app.py`
(new routes for in-place edit + event stream append).

======================================================================
v1.4.0 — BULK OPERATIONS
======================================================================

The bulk-actions tile on the queue exists but is currently a
placeholder. Wire the real actions:

  - bulk approve (advance approval step)
  - bulk assign operator
  - bulk schedule
  - bulk reject
  - bulk export

Each action runs per-row through the same permission gate the
single-row path uses; failures are reported in aggregate via the
toast system. The first tangible payoff from the tile architecture.

Files: `templates/schedule.html`, `app.py` (`schedule_bulk_actions`
route), `static/styles.css` (bulk-actions tile reveal animation).

======================================================================
v1.5.0 — SEARCH
======================================================================

SQLite FTS5 virtual table over `sample_name`, `description`,
`request_no`, and requester name. New search box on the queue and
dashboard; results link into the existing card view.

  - FTS5 ships with SQLite, so no new dependency.
  - Index rebuilt by `init_db()` if missing.
  - Trigger on `sample_requests` INSERT/UPDATE keeps it current.

Files: `app.py` (init_db, new search route), `templates/schedule.html`,
`templates/dashboard.html`.

----------------------------------------------------------------------
NOT DOING (architectural decisions)
----------------------------------------------------------------------

Listed here so future agents do not spin them up speculatively.

  - Python 2 compatibility — no.
  - Server-side rendering of charts — Chart.js is fine.
  - Migrations framework (alembic / yoyo) — `init_db()` handles
    schema evolution adequately for SQLite.
  - Background job queue (celery / rq) — no async work justifies
    the dependency yet.
  - WebSockets — polling at the existing intervals is fine.
  - Build step / bundler / transpiler — vanilla JS + a single CSS
    file is the design rule.
  - Ollama / local LLM offload — dropped at v1.3.0. The Mac mini
    is a production host, not a compute bridge.
