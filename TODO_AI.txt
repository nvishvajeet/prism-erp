PRISM — Active Plan
====================

Read this before starting work. It is intent, not contract.

Guidelines for AI agents
------------------------
1. Verify state before acting. Items may already be done.
2. Fix root causes, not symptoms.
3. If you find bugs not listed here, fix them and note them.
4. Don't force architectural changes unless they unblock the current task.
5. Commit and push every ~2 minutes. Never leave changes hanging.
   - Default rhythm is: pull → work → commit → push.
   - Commit each file change as it lands unless a single logical unit
     genuinely spans multiple files. Small, focused commits > large ones.
   - `git push` is mandatory after every commit. No local-only commits.
   - If you are mid-refactor and something breaks, commit the WIP under
     a `wip:` prefix and push — never sit on broken work.
6. Keep the visibility audit + populate crawl green before every push.
   `crawler_suite.py` (see `crawlers/` module) exposes `run all` for
   batched verification.

----------------------------------------------------------------------
DESIGN PHILOSOPHY (load-bearing — read every session)
----------------------------------------------------------------------

Apple / Jony Ive / Ferrari. Every element earns its place.

  - **Tiles, not pages.** Each concern is a self-contained widget tile
    on a fluid grid. No mixed concerns inside one card.
  - **Pagination, never scroll.** Long lists use `paginated_pane`. No
    `overflow: auto` inside content panes.
  - **Macros over markup.** If a pattern shows up twice, it becomes a
    macro. Templates compose; they do not duplicate.
  - **One queue, one card.** A request is the same object everywhere
    it appears — same identity, same status block, same actions.
  - **Visibility is sliced server-side first.** `data-vis` is a safety
    net, not the gate.
  - **Empty space is content.** Cramped is the failure mode, not bold.

Reference implementation: `templates/instrument_detail.html` — 10 tiles
on a 6-column fluid grid. CSS at `static/styles.css` under "INSTRUMENT
DETAIL — TILE ARCHITECTURE". This is the gold standard. Match its rhythm.

PHASE 5 — WIDGET PROPAGATION (current focus)
----------------------------------------------------------------------

Goal: every user-facing page reaches the polish level of
instrument_detail.html. Extract recurring widgets to macros so the
codebase teaches itself the pattern.

Order matters: extract macros first, then convert pages, so each page
benefits from the macros built before it.

W5.1  Shared widget macros (foundation — do these first)
  Each macro lives in `_page_macros.html` unless noted.

  - status_pills_row(pills, active, on_change)
      Unifies .stream-pill (schedule), .role-toggle (calendar),
      .warroom-pill (stats) into one filter pill component.
      Canonical class: .filter-pill. Deprecate the others in the same
      pass.

  - queue_action_stack(row, operators, can_accept, can_assign)
      The Accept Sample + Quick Assign forms currently inlined into
      schedule.html, instrument_detail.html, and dashboard.html.
      Three call sites become one.

  - person_chip(user, link=True, avatar=True, size='md')
      Avatar circle + name (+ optional link). Replaces ad-hoc team
      lists in instrument_detail, request_detail, dashboard, stats.

  - activity_feed(entries, pane_id, page_size=6, threaded=False)
      Wraps the timeline pattern. `threaded=True` enables left/right
      message alignment for request_detail conversation view.

  - metadata_grid(items, compact=False)
      <dl> grid for label/value pairs. Replaces .detail-grid,
      .compact-detail-grid, .tile-info-meta with one canonical list.

  - kpi_grid(items, variant='card')
      KPI counters. Replaces .stats-kpi-row (finance),
      .grid-auto-stats (stats), and stat blob clusters on dashboard.

  - approval_action_form(step, allow_file=False, allow_note=True)
      Approve/Reject form for one approval step. Currently lives only
      inside request_detail.html; extract so other pages can host it.

  - toggleable_form(form_id, trigger_label, body)
      One disclosure pattern to replace ad-hoc <details>, jQuery
      toggles, and modal divs.
