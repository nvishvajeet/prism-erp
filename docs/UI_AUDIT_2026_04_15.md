## Methodology

1. **POST form enumeration.** `/tmp/form_audit.py` (run from
   `.venv/bin/python`) walks every `templates/**/*.html`, extracts
   `<form … method="post"> … </form>` spans, and reports:
   - presence of `<button type=submit>` / `<input type=submit>` /
     bare `<button>` (HTML default type IS submit);
   - count of `<input>` / `<select>` / `<textarea>` fields;
   - label coverage (by `<label for=id>` — does not catch the
     label-wrap pattern, flagged manually instead).
2. **Grid-class enumeration.** Every `class="*-tiles*"` in templates,
   cross-referenced against `static/styles.css` to find
   missing-override grids.
3. **CSS clipping hunt.** `grep` for `overflow: hidden`,
   `max-height`, and `display: none` patterns in templates and
   styles.css; each hit classified as "intentional chrome",
   "server-side truncation", or "actual clip".
4. **Spot checks.** `vendor_form.html`, `new_request.html`,
   `finance_grant_detail.html` (the three forms with the highest
   field count) manually read to confirm the label-wrap pattern.

## Findings

### F-01 · `insights-tiles` grid missing `.tile` override

**Severity:** medium-high (silent content clipping on four pages).

**Where:** `templates/insights_list.html`, `templates/insights_detail.html`, `templates/insights_form_control.html`, `templates/insights_new.html` — all four use `<section class="insights-tiles">`.

**Root cause.** `static/styles.css:3781` sets `.tile { overflow: hidden }` as the base rule; `static/styles.css:3824` caps `.tile { max-height: 600px }` as a defensive "don't blow up the grid" default. Every other grid container provides an override:
```
.inst-tiles > .tile            → max-height:none; overflow:visible
.instruments-tiles > .tile     → max-height:none; overflow:visible
.request-tiles > .tile         → max-height:none; overflow:visible
.sitemap-tiles > .tile         → max-height:none; overflow:visible
.new-request-tiles > .tile     → max-height:none; overflow:visible
.finance-tiles > .tile         → max-height:none
.admin-notices-tiles > .tile   → max-height:none
(etc. — see styles.css:3834..3895)
```
`.insights-tiles > .tile` — missing. So any insights tile that overflows 600px is silently clipped.

**Fix:** added to `static/css/ui_audit_2026_04_15.css`. Pattern copies the `inst-tiles` case (entity detail: content first, no forced scroll).

**Verification (Phase 3):** load each of the 4 insights pages at desktop width, confirm no visible clipping on tiles with long content. If the activity feed on `insights_detail.html` has >600px of entries, the fix is required.

### F-02 · View-pane data-entry overflow fallback (defensive)

**Severity:** low (no currently-clipping template found).

**Why add it.** The sprint scope says "view panes should have wrap if they are showing data entry items currently they are overflowing." No template in the audit actually hits this — every form page uses a grid class that's already uncapped. But the base `.tile` rule still caps to 600px, and nothing prevents a future tile from landing inside `stats-tiles` (320px cap) or `inbox-tiles` (600px cap) with a `.form-grid` inside. Adding a preventative rule.

**Rule.** `.tile .form-grid`, `.tile .action-form` → `min-height: 0; overflow-y: auto`.

**Why `min-height: 0`.** `.tile` is `display: flex; flex-direction: column`; flex children default to `min-height: auto` in column layouts, which silently prevents `overflow-y` from taking effect. The `min-height: 0` unlocks it.

---

### F-03 · Long-text value wrap on metadata / data rows

**Severity:** low.

**Why.** `word-break: break-word` is applied to a few specific components in styles.css (line 2030, 2951, 3985) but NOT universally on `metadata-grid`'s value column. A long URL / hash / unbroken string in a metadata row will push the cell wider than its grid column, which on width-capped tiles (`dashboard-tiles`, `stats-tiles`) gets clipped.

**Rule.** `.metadata-grid .meta-value`, `.metadata-grid dd` → `overflow-wrap: anywhere; word-break: break-word`.

---

### F-04 · Form action-row mobile wrap

**Severity:** low.

**Why.** Several templates render the end-of-form Save/Cancel pair as an inline flex row without `flex-wrap`. On <375px viewports the Cancel button overflows to the right.

**Rule.** `.form-grid .form-actions`, `.action-form .form-actions`, and any `[class*="actions-row"]` inside a `.form-grid` → `display:flex; flex-wrap:wrap; gap:0.5rem`.

## No-fix decisions

### F-NF1 · `portfolio.html` · `#todayBuyForm` is JS-submit

`templates/portfolio.html:72` opens `<form id="todayBuyForm" method="post">` but has no visible submit button. The action is `<button type="button" class="link-button subtle" id="buyOneOfEach">✓✓ Buy one of each</button>` plus row checkboxes; both wire into a JS handler (see `<script>` below the form) that submits the form via `form.requestSubmit()`.

Intentional: "tick a row's checkbox to log that fund at the recommended amount" is the primary UX; "Buy one of each" is the bulk shortcut. Converting to a classic `<button type="submit">` would either confuse the checkbox flow or duplicate it.

**Recommendation:** leave as-is. Document in `docs/PHILOSOPHY.md` if/when we codify a "JS-submit vs button-submit" convention.

### F-NF2 · Field labels (label-wrap vs `<label for=>`)

Spot-check confirms every form uses `<label>Text <input …></label>` (label-wrap) rather than `<label for="id">Text</label> <input id="id">`. Both patterns are valid per the HTML spec; the label-wrap pattern does not require `for=`/`id=` association and is already accessible to screen readers. The audit script reports high "fields without labels" counts only because it matches against `for=`; that's a script limitation, not a template bug.

**Recommendation:** ignore script false positives. If we want a zero-false-positive label crawler, add it as a `behavioral` wave with proper label-wrap detection.
