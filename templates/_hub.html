Tile Grid Layout System (6-column fluid)
Pattern: All content pages use a 6-column fluid grid with semantic tile regions.

display: grid; grid-template-columns: repeat(6, minmax(0, 1fr)); gap: 1rem;

Three live implementations:
  • .inst-tiles — 10 tiles (instrument detail). Reference implementation.
  • .request-tiles — 8 tiles (request detail). Adds 4-col breakpoint at 1100px.
  • .new-request-tiles — 2 tiles (new request form). Full-width.
Rule: Any page with >1 semantic region gets a tile grid. Single forms exempt.

Backend Patterns to Reuse
Auth decorators: @login_required, @role_required(*roles), @owner_required, @instrument_access_required(level), @rate_limit(max, window)

Scope SQL: request_scope_sql(user, alias) — role-aware WHERE clause generator. Every list query uses this.

State machine: REQUEST_STATUS_TRANSITIONS dict + assert_status_transition(current, target) enforcer.

Audit chain: log_action(actor, entity_type, entity_id, action, payload) — SHA-256 linked, immutable.

Card policy: request_card_policy(user, request) → returns fields and actions sets per (user, request) combination.

Visibility: Two-layer — server-side (card_policy) + client-side (data-vis="{{ "{{ V }}" }}" on every element).
Starter Pack — Data Pipeline Pattern (from Portfolio Plan)
Linear Pipeline Architecture
Pattern: Each script reads inputs (YAML config + prior JSON), computes, writes one *_state.json. No circular dependencies. Shell script enforces ordering.

9 steps in portfolio-plan:
  1. fetch_nav.py → raw data (fund NAVs, FX, indices)
  2. market.py → z-scores, 200DMA, breadth signals
  3. macro.py → CPI, repo rate, GSec yields
  4. news.py → RSS sentiment (optional)
  5. analyze.py → portfolio state, drift, rebalance
  6. simulate.py → Monte Carlo (28y wealth projection)
  7. horizon.py → monthly outlook + conviction score
  8. daily.py → today's buy recommendation
  9. peer_compare.py → fund rankings vs peers
Orchestrator: run.sh (~15–30s total)
Config: portfolio.yaml — single source of truth for holdings, weights, rates, settlement rules
Flask integration: app.py reads JSON state files directly — no database layer needed

AI Agent Rules — Read Before Starting Any Project
12 Workflow Rules
  1. Read what you need, not everything. CLAUDE.md auto-loads. Read policy files only when needed.
  2. Pull → work → test → commit → push. No local-only commits.
  3. Run pre-commit gate before every commit. Each project defines its own gate in WORKFLOW.md.
  4. Fix root causes, verify state before acting.
  5. Commit frequently but not excessively. One commit per meaningful unit of work.
  6. Batch terminal permissions, front-load shell operations.
  7. Full gates only at release boundaries. Mid-flight = smoke/sanity only.
  8. Read project's reusable abstractions before adding code.
  9. Hard attributes locked. Changes require major version bump.
  10. Website stays up. Atomic deploys. Never force-push stable branch.
  11. Log rare actions with ai-log. Credential changes, infra changes, cross-project actions only.
  12. Cross-project auto-tasks — one check per local day.
Git Hard Rules
  • NEVER open a bare repo in Sourcetree or any GUI client.
  • ONE bare per project under ~/.claude/git-server/.
  • Clone from Level 1 LOCAL bare, never from Level 2 upstream.
  • Bares MUST NOT be in Dropbox, iCloud, or any synced folder.
  • Push to origin (Level 1) always. Level 2 push needs a reason.
  • Never force-push or rewrite history on a release branch.
  • Imperative commit subject ≤ 70 chars, body explains why.
Onboard a New Project
Requires explicit user approval. The registry is the gate.

~/.claude/git-server/bin/new-project.sh <name> [existing-working-copy]

Creates Level-1 bare, installs hook, resets origin, seeds WORKFLOW.md stub.
For Level-2 (mini mirror): --mirror-to-mini flag.
After: update registry in AGENT_POLICY.md §1 and git-server/README.md §3.
