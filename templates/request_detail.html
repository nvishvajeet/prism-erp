{# ── TILE: sample-approval explainer — moved to top for visibility ── #}
{{ card_heading("", ("How does lab review work?" if _lab_mode else "How does sample approval work?")) }}
Show the full pipeline
    {% if _lab_mode %}
  1. Created. A sample request enters the lab queue and becomes visible on the workflow strip above.
  2. Lab review. The Dean or Conductor reviews whether the request is ready to move and whether the right instrument profile owns it.
  3. Instrument assignment. The request is assigned to the person or machine lane responsible for that instrument, and the sample is marked received once it reaches the lab.
  4. Run + results. The instrument team carries out the work, attaches files, and records the result trail on this page.
  5. Completed. The request becomes the permanent audit record for that sample. Reopening should be rare and deliberate.
    6. {% else %}
  6. Submitted. The requester creates the request and the lifecycle strip above shows it on the left. Nothing is approved yet.
  7. Finance review. The finance admin confirms the grant / account code. If the instrument is charged per use, the rate is locked at this step.
  8. Professor approval. The requester's supervising professor signs off on the scientific intent. This step can be skipped if the requester already holds operator-level access.
  9. Instrument admin schedule. The operator group owning the instrument picks a slot, marks the sample received, and queues the job. Only users in the operator group for this instrument see this step.
  10. Run + results. The operator runs the sample, attaches result files to the Activity tile above, and marks the request complete. The audit chain is updated and the queue advances.
  11. Completed. Requester, professor, finance, and operators can all see the result files. A completed request is immutable except for a flagged re-open by the owner role.
    12. {% endif %}
